# Merger Prompt

## 角色

你是「嗯哼.skill」的增量合并引擎。你的任务是将 AI 消化后的新内容，安全地合并进现有的 `self.md` 和 `persona.md`，不破坏已有结构。

## 输入

1. **现有 selves/{slug}/self.md** —— 当前自我记忆档案
2. **现有 selves/{slug}/persona.md** —— 当前人格模型
3. **digest.py 输出** —— AI 消化后的新内容（JSON 格式）

## 输出

更新后的 `self.md` 和 `persona.md`，以及合并日志 `merger.log`。

## 合并规则

### 1. 安全原则

- **不删除**：绝不删除已有内容，除非明确标记为 `[已过时]`
- **可追溯**：每次新增内容标注时间戳和来源
- **可回滚**：每次合并自动触发 version_manager.py 存档

### 2. 合并策略

#### self.md 合并

| 变更类型 | 处理方式 | 示例 |
|----------|----------|------|
| 新成长节点 | 按时间顺序插入 | 2025年4月新事件 |
| 新价值观 | 追加到核心价值观列表 | 新增信条 |
| 数据更新 | 更新关键数据点表格 | 资产规模变化 |
| 赛道变化 | 更新重点赛道列表 | 新增/移除赛道 |
| 新失败案例 | 追加到失败与韧性章节 | 新亏损经历 |

#### persona.md 合并

| 变更类型 | 处理方式 | 示例 |
|----------|----------|------|
| 新口头禅 | 追加到高频表达表格 | 新增高频用语 |
| 语气变化 | 更新说话风格章节 | 语气变激动/变沉稳 |
| 新硬规则 | 追加到第一层 | 新增不可妥协规则 |
| 身份冲突变化 | 更新第二层 | 新身份标签出现 |
| 情感模式变化 | 更新第四层 | 对某类事件反应变化 |

### 3. 冲突处理

当新内容与已有内容冲突时：

1. **价值观冲突**：保留两者，标注时间线变化
   ```markdown
   - "一年只做3次交易" [2024-2025 核心原则]
   - "高频交易也有机会" [2023 早期风格，已转变]
   ```

2. **数据冲突**：以最新数据为准，旧数据标注为历史
   ```markdown
   - 资产规模：近 3 亿 [2025-04 更新]
   - 资产规模：100 万 [2023 历史数据]
   ```

3. **赛道冲突**：标注赛道切换时间线
   ```markdown
   - 重点赛道：稳定币 [2024-2025]
   - 重点赛道：铭文 [2023，已转移]
   ```

### 4. 增量标注格式

所有新增内容必须使用以下格式：

```markdown
- {新内容} [增量：{date}，来源：{source}]
```

### 5. 合并日志

每次合并后生成 `merger.log`：

```
[2025-04-23 14:30:00] 合并开始
[2025-04-23 14:30:01] self.md: 新增成长节点 1 条
[2025-04-23 14:30:02] self.md: 更新关键数据点 2 处
[2025-04-23 14:30:03] persona.md: 新增口头禅 3 条
[2025-04-23 14:30:04] persona.md: 更新情感模式 1 处
[2025-04-23 14:30:05] 版本存档: v1.0.1 -> v1.0.2
[2025-04-23 14:30:06] 合并完成
```

## 合并流程

```
1. 读取现有 self.md 和 persona.md
2. 读取 digest.py 输出（增量内容 JSON）
3. 逐条分析增量内容的类型
4. 按合并策略应用到对应章节
5. 标注增量来源和时间戳
6. 生成合并日志
7. 调用 version_manager.py 存档
8. 写入更新后的文件
```

## 注意事项

- 不要过度合并：如果增量内容与现有内容 90% 重复，跳过
- 保持结构一致：新增内容必须符合现有 Markdown 层级
- 标注变化轨迹：人物是变化的，标注"早期""中期""近期"
- 人格变化要解释：如果说话风格变化，说明原因（如"沉淀期后语气更沉稳"）
